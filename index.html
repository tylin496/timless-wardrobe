    Where this data comes from

    Each reload builds one merged wardrobe list: rows you added are merged first, then the catalogue (from Supabase or the bundled seed), with per-piece edits applied the same way whether the piece started as seed or custom. Cover and gallery URLs are plain strings on each row (often https://… public object URLs).

    • Catalogue source — With js/tw-supabase-config.js and a non-empty wardrobe_items fetch, that table is the catalogue. Otherwise the app uses data/wardrobe.js (WARDROBE_ITEMS).
    • Your wardrobe in the browser — Extra rows and any edits not yet in the cloud live under localStorage keys such as timeless-wardrobe-custom-items-v1, timeless-wardrobe-archive-overrides-v1, and timeless-wardrobe-archive-hidden-v1, plus optional data/custom-items.json when you use npm run dev.
    • Supabase sync (same bar for every piece) — With cloud configured, new pieces and edits are written to wardrobe_items (catalogue pieces you changed are mirrored there like any other row). Hidden archive ids and any legacy per-id override blobs still round-trip through wardrobe_app_state row id = default when needed.
    • Saved outfits — Supabase tables outfits + outfit_items when the wardrobe is cloud-backed and outfit fetch succeeds; otherwise localStorage key timeless-wardrobe-outfits-v1.

    — full snapshot: custom rows, catalogue overrides, hidden ids, outfits, and display prefs (keep this file somewhere safe when you are not on cloud sync).

    Required: brand, name, section, and record type. Photos are optional: in one batch, the first file becomes the cover and any further files are saved to the gallery (you can change them when editing). Optional fields and extra photos help later when you search or edit. Run npm run dev and open the app from that server so new pieces are written straight to data/custom-items.json in the project (and still cached in the browser). Plain static hosting only keeps a browser copy unless you use cloud sync or export JSON.
